# rslive 项目概览与架构设计

## 项目定位

**rslive** 是一个全面的 Rust 实时流媒体协议库，专注于高性能、低延迟的多媒体处理和传输。目标是成为 Rust 生态系统中流媒体基础设施的首选库。

## 核心目标

1. **高性能**: 利用 Rust 的零成本抽象和无 GC 特性，实现接近裸机性能
2. **多协议支持**: 原生支持 RTMP、SRT、WebRTC、HLS、DASH 等主流协议
3. **模块化设计**: 协议层、编解码层、传输层完全解耦
4. **生产就绪**: 提供完整的错误处理、监控指标、热重载等企业级特性

## 技术选型

### 异步运行时: Tokio
**原因**:
- Rust 生态最成熟的异步运行时
- 多线程 work-stealing 调度器
- 零成本抽象，接近裸机性能
- 丰富的生态系统（tokio-util, tokio-stream 等）

### 网络 I/O: io_uring (Linux) + Tokio (跨平台)
**策略**:
- Linux 使用 tokio-uring 获得极致性能
- 其他平台使用标准 Tokio
- 抽象层屏蔽底层差异

### 序列化: Bytes + byteorder
**原因**:
- Bytes crate 提供零拷贝缓冲区共享
- 引用计数避免不必要的数据复制
- 与 Tokio 生态完美集成

### 并发原语: parking_lot + crossbeam
**原因**:
- parking_lot 比标准库 Mutex 更快
- crossbeam 提供无锁数据结构

## 关键设计决策

### 1. 零拷贝架构
- 使用 `Bytes` 和 `BytesMut` 作为核心缓冲区类型
- 协议解析器直接操作原始缓冲区，避免 memcpy
- 引用计数机制允许安全共享数据

### 2. 无锁设计
- 使用 channel 进行组件间通信（MPMC: crossbeam, SPSC: flume）
- 避免在热路径使用 Mutex
- 必要时使用 parking_lot 的轻量级锁

### 3. 内存池化
- 预分配缓冲区池减少 allocator 压力
- 对象池复用频繁创建/销毁的对象
- 批量处理减少系统调用

### 4. Backpressure 机制
- 每个组件暴露背压信号
- 自动流控防止内存溢出
- 可配置的缓冲策略

## 参考项目

### SRS (Simple Realtime Server)
- **优势**: State Threads 实现的高并发、成熟稳定
- **可借鉴**: 单进程架构、配置文件热加载、HTTP API 设计

### MediaMTX
- **优势**: 单二进制零依赖、多协议自动转换
- **可借鉴**: 模块化协议设计、配置热重载

### SRT Protocol
- **优势**: 超低延迟、抗丢包、现代加密
- **可借鉴**: ARQ 机制、FEC 算法、拥塞控制

### Bun
- **优势**: 极致启动速度、低内存占用
- **可借鉴**: 最小化依赖、编译期优化
