# 流媒体协议设计哲学

## 设计者的思考

想象你是 2000 年代初的 Adobe 工程师，面对的是这样一个世界：

- 网络带宽有限（拨号上网还在普及）
- Flash Player 需要安装在 95% 的电脑上
- 直播和点播需求并存
- 防火墙无处不在

每个协议的设计都是**特定时代背景下的最优解**。

---

## RTMP：为低延迟而生

### 设计背景

**问题**：如何让 Flash Player 实时播放视频？

**时代背景（2002年）**：
- TCP 是唯一可靠的选择
- UDP 被防火墙广泛封锁
- HTTP 不支持长连接
- 需要传输命令、音频、视频多种数据

### 核心设计决策

#### 1. 为什么用 TCP 而不是 UDP？

```
设计者的思考：
┌─────────────────────────────────────────────────────────┐
│ UDP 方案：                                               │
│   ✓ 延迟极低                                             │
│   ✗ 防火墙阻断率高（企业网络、NAT）                        │
│   ✗ 需要自己实现拥塞控制、重传                            │
│   ✗ 开发成本高，兼容性差                                  │
│                                                         │
│ TCP 方案：                                               │
│   ✓ 防火墙穿透率高（80/443/1935 端口）                    │
│   ✓ 系统原生支持，稳定可靠                                │
│   ✗ 延迟较高（Nagle 算法、ACK 延迟）                      │
│                                                         │
│ 决策：选择 TCP，用 Chunk 分块来解决延迟问题                │
└─────────────────────────────────────────────────────────┘
```

#### 2. 为什么要设计 Chunk 分块？

**没有分块时的问题**：
```
场景：正在传输一个 10MB 的关键帧

时间线：
0s ────────────────────────────────────────────── 0.5s
    [====== 传输 10MB 视频帧 ======]
    
此时：音频帧到达，但必须等待视频帧传完才能发送！
延迟：500ms+
```

**分块后的改进**：
```
Chunk Size = 128 字节

时间线：
0s ────┬────┬────┬────┬────┬────┬────┬──── 0.02s
       │视频│音频│视频│音频│视频│音频│...
       │C1  │C1  │C2  │C2  │C3  │C3  │

音频帧可以"插队"，延迟降低到 20ms！
```

**设计者的智慧**：
> "既然 TCP 是串行传输，那就把大消息切成小块，让紧急数据有机会'插队'。"

#### 3. 为什么握手这么复杂？

```
C0/C1 → S0/S1 → C2 → S2（共 6 个包，1537+ 字节）

为什么不简单点？比如 TCP 三次握手就够了？

设计者的考虑：
1. 时间同步：C1 中的时间戳用于后续计算延迟
2. 防重放攻击：随机数据验证对方身份
3. 协议版本协商：C0/S0 确保双方版本兼容
4. 带宽探测：1536 字节的握手可以粗略估计带宽
```

### RTMP 的优势与劣势

| 优势 | 劣势 |
|------|------|
| 延迟低（2-5秒） | 需要专用服务器（1935端口） |
| 防火墙穿透好（可走 80/443） | 非标准协议，被 CDNs 限制 |
| 支持双向通信 | 已被 HTML5 时代抛弃 |
| 分块设计精妙 | 实现复杂度高 |

## FLV：为存储和回放而生

### 设计背景

**问题**：RTMP 流如何保存为本地文件？

### 核心设计决策

#### 1. 为什么用 Tag 结构？

```
设计者的思考：

方案 A：直接存储 TCP 流
┌─────────────────────────────────────────────┐
│ 问题1：无法 seek（不知道每帧的位置）          │
│ 问题2：无法倒放（没有索引）                   │
│ 问题3：不知道有没有音频/视频                  │
└─────────────────────────────────────────────┘

方案 B：Tag 结构
┌─────────────────────────────────────────────┐
│ FLV Header → 我有什么内容                     │
│ Tag 1 → 时间戳 + 类型 + 数据                  │
│ Tag 2 → 时间戳 + 类型 + 数据                  │
│ ...                                         │
│ Previous Tag Size → 可以反向解析！           │
└─────────────────────────────────────────────┘
```

#### 2. 为什么要 Previous Tag Size？

```rust
// 正向解析：从文件头开始
Header → Tag1 → Tag2 → Tag3 → ...

// 反向解析：从文件尾开始
... ← Tag3 ← Tag2 ← Tag1 ← Header
      ↑
   通过 Previous Tag Size 定位
```

**应用场景**：
- 视频快进/快退
- 从任意位置开始播放
- 损坏文件的恢复

#### 3. 为什么时间戳用 24+8 位？

```
设计者的计算：
- 视频时长 < 4.6 小时：24 位够用（16,777,215 ms）
- 视频时长 > 4.6 小时：需要扩展（直播场景）

24 位基础 + 8 位扩展 = 32 位
支持最长：49.7 天

为什么不直接用 32 位？
答：向后兼容！老播放器只读 24 位也能播放短视频。
```

### FLV 的优势与劣势

| 优势 | 劣势 |
|------|------|
| 结构简单，易于解析 | 只支持 Flash 相关编码 |
| 支持 seek 和倒放 | 已被 MP4 取代 |
| 与 RTMP 天然兼容 | 无法支持多轨道 |
| Previous Tag Size 支持反向解析 | 元数据有限 |

## HLS：为兼容性而生

### 设计背景

**问题**：如何在 iOS 上播放流媒体？

**时代背景（2009年）**：
- iPhone 不支持 Flash
- Apple 需要自己的流媒体方案
- HTTP 是唯一被广泛支持的协议
- 防火墙、代理、CDN 都对 HTTP 友好

### 核心设计决策

#### 1. 为什么选择 HTTP？

```
设计者的权衡：

RTMP：
┌─────────────────────────────────────────────┐
│ ✗ 需要专用服务器                             │
│ ✗ 防火墙可能阻断                             │
│ ✗ CDN 支持有限                               │
│ ✓ 延迟低                                     │
└─────────────────────────────────────────────┘

HTTP：
┌─────────────────────────────────────────────┐
│ ✓ 任何 Web 服务器都能托管                    │
│ ✓ 防火墙 100% 放行                           │
│ ✓ CDN 原生支持                               │
│ ✓ 穿透 NAT、代理                             │
│ ✗ 延迟高                                     │
└─────────────────────────────────────────────┘

Apple 的选择：牺牲延迟，换取最大兼容性
```

#### 2. 为什么用 M3U8 文本格式？

```
设计者的思考：

方案 A：二进制格式
┌─────────────────────────────────────────────┐
│ 优点：体积小，解析快                          │
│ 缺点：不可读，调试困难                        │
└─────────────────────────────────────────────┘

方案 B：文本格式（M3U8）
┌─────────────────────────────────────────────┐
│ 优点：                                       │
│   - 可读性强，curl 直接看                    │
│   - 任何编辑器都能修改                       │
│   - 解析器实现简单                           │
│   - 继承自 M3U 音乐列表，兼容 Winamp/iTunes  │
│ 缺点：体积稍大                               │
└─────────────────────────────────────────────┘

决策：选择文本格式，简单至上
```

#### 3. 为什么分片是 6-10 秒？

```
设计者的计算：

分片时长 = 关键帧间隔 × N

关键帧间隔：
- 30fps 视频，每 2 秒一个关键帧 → 60 帧
- 60fps 视频，每 2 秒一个关键帧 → 120 帧

延迟组成：
┌─────────────────────────────────────────────┐
│ 分片生成时间（6秒）                           │
│ + 播放列表更新延迟（0-6秒）                   │
│ + 下载时间（网络速度）                        │
│ + 缓冲时间（1-2个分片）                       │
│ ───────────────────────────                  │
│ 总延迟 ≈ 15-30 秒                            │
└─────────────────────────────────────────────┘

为什么不能更短？
- 关键帧间隔限制
- 分片太多，播放列表更新频繁
- CDN 缓存效率下降
```

#### 4. 为什么支持多码率？

```
Apple 的愿景：

用户的网络环境千差万别：
┌─────────────────────────────────────────────┐
│ WiFi 用户：希望看 1080p                      │
│ 4G 用户：希望看 720p                         │
│ 电梯/地铁：只能看 480p                       │
└─────────────────────────────────────────────┘

解决方案：
Master Playlist → 列出所有码率选项
客户端根据网络状况自动切换
切换时不需要重新建立连接！

这是 HTTP 的巨大优势：每个分片独立下载
```

### HLS 的优势与劣势

| 优势 | 劣势 |
|------|------|
| 100% 防火墙穿透 | 延迟高（15-30秒） |
| CDN 原生支持 | 不适合实时互动 |
| 多码率原生支持 | 服务端需要维护分片 |
| iOS 原生支持 | 分片管理复杂 |
| 可用普通 Web 服务器 | 首屏加载慢 |

## AMF0：为简单可靠而生

### 设计背景

**问题**：如何在 Flash Player 和服务器之间传输结构化数据？

**时代背景（2002年）**：
- XML 太冗余
- JSON 还不流行
- 需要支持 Flash 特有类型（MovieClip、Reference）

### 核心设计决策

#### 1. 为什么用类型标记？

```
设计者的思考：

方案 A：像 JSON 一样推断类型
┌─────────────────────────────────────────────┐
│ 问题：                                       │
│   - "123" 是字符串还是数字？                  │
│   - true/false 怎么表示？                     │
│   - Date 类型怎么处理？                       │
└─────────────────────────────────────────────┘

方案 B：显式类型标记
┌─────────────────────────────────────────────┐
│ 0x00 + 8字节 → 这是数字                       │
│ 0x01 + 1字节 → 这是布尔                       │
│ 0x02 + 长度 + 内容 → 这是字符串               │
│                                              │
│ 解析器不需要猜测，直接读类型                   │
└─────────────────────────────────────────────┘

决策：显式标记，避免歧义
```

#### 2. 为什么 Reference 要独立类型？

```
场景：传输对象数组

[
  {name: "张三", age: 30},
  {name: "张三", age: 30},  // 同一个对象
  {name: "张三", age: 30}   // 同一个对象
]

方案 A：每次完整传输
┌─────────────────────────────────────────────┐
│ 传输量：3 × 30 字节 = 90 字节                 │
└─────────────────────────────────────────────┘

方案 B：引用机制
┌─────────────────────────────────────────────┐
│ 第一次：完整对象（30字节）                    │
│ 第二次：Reference(0)（3字节）                 │
│ 第三次：Reference(0)（3字节）                 │
│ 传输量：30 + 3 + 3 = 36 字节                  │
│ 压缩率：60%                                  │
└─────────────────────────────────────────────┘
```

## AMF3：为效率而生

### 设计背景

**问题**：AMF0 太冗余，如何优化？

### 核心设计决策

#### 1. 为什么用 U29 变长整数？

```
设计者的观察：

大部分整数值都很小（< 128）

AMF0 的做法：
┌─────────────────────────────────────────────┐
│ 所有整数都用 8 字节 Double                   │
│ 值 1 → 9 字节（标记 + 8字节）                │
│ 值 100 → 9 字节                              │
│ 值 1000000 → 9 字节                          │
└─────────────────────────────────────────────┘

AMF3 的优化：
┌─────────────────────────────────────────────┐
│ 值 1 → 2 字节（标记 + 1字节 U29）             │
│ 值 100 → 2 字节                              │
│ 值 1000000 → 9 字节（超过 29 位，用 Double）  │
│                                              │
│ 小整数节省 7 字节！                          │
└─────────────────────────────────────────────┘
```

#### 2. 为什么引入字符串引用表？

```
设计者的观察：

RTMP 命令中，字符串大量重复：
┌─────────────────────────────────────────────┐
│ "onStatus"                                   │
│ "NetStream.Play.Start"                       │
│ "level"                                      │
│ "code"                                       │
│ "description"                                │
└─────────────────────────────────────────────┘

AMF0：
每次完整传输，重复浪费

AMF3：
┌─────────────────────────────────────────────┐
│ 第一次："onStatus" → 存入表，索引 0          │
│ 后续：直接用索引 0 引用                       │
│                                              │
│ 传输量减少 80%+                               │
└─────────────────────────────────────────────┘
```

## 总结：设计者的智慧

1. **RTMP**：用复杂性换取低延迟，分块设计是精髓
2. **FLV**：用简单结构换取可 seek 性，Previous Tag Size 是亮点
3. **HLS**：用延迟换取兼容性，HTTP + 文本格式是关键
4. **AMF0**：用显式类型换取可靠性，避免解析歧义
5. **AMF3**：用引用表换取效率，变长编码节省空间

**每个设计都是特定约束下的最优解**，没有绝对的好坏，只有适合不适合。
